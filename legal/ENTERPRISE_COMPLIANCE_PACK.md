## 1. B-BBEE CERTIFICATE

### Your Classification: EME (Exempted Micro Enterprise)

**Turnover under R10M = EME. This is the easiest path.**

| Ownership | Automatic Level | Procurement Recognition |
|-----------|----------------|------------------------|
| Any ownership | Level 4 | 100% |
| 51%+ black ownership | Level 2 | 125% |
| 100% black ownership | Level 1 | 135% |

**Why this matters for Netcare**: Netcare is Level 3 B-BBEE. They get bonus procurement points for using higher-level suppliers. Level 1-2 makes you a preferred vendor.

### How to Get Your Certificate (Choose One)

**Option A: CIPC Certificate (Free, recommended)**
1. Go to https://eservices.cipc.co.za
2. Log in with CIPC credentials
3. Apply for B-BBEE Certificate
4. Certificate issued electronically at no cost
5. Valid for 12 months — renew annually

**Option B: Sworn Affidavit (Free, alternative)**
1. Download template: https://www.cipc.co.za/wp-content/uploads/2023/01/Affidavit-EME-Gen.pdf
2. Complete turnover and ownership details
3. Sign in person at SAPS (Commissioner of Oaths — free)
4. Rules:
   - Deponent and Commissioner sign on the SAME date
   - Signatures must be physical (not electronic)
   - Commissioner cannot be an employee or have interest in the company

## 2. TAX CLEARANCE CERTIFICATE

### Process (Free, 1-48 hours)

1. Log in to SARS eFiling: www.sarsefiling.co.za
2. Navigate: Tax Compliance Status → Request
3. Select type: **"Good Standing"** (for procurement)
4. Submit request
5. If compliant, TCC issued immediately with a verification PIN
6. Netcare can verify your TCC using the PIN

**Prerequisites**: All tax returns filed, no outstanding SARS debt.

### 4.3 Information Security Policy

```
VISIOCORP INFORMATION SECURITY POLICY
Version: 1.0
Classification: Internal
Owner: Information Officer

1. PURPOSE
To establish the security standards for protecting personal information
and health data processed through the Netcare Health OS platform, in
compliance with POPIA section 19 and the Health Information Regulations
2026.

2. SCOPE
All systems, data, personnel, and processes involved in the operation
of the Netcare Health OS platform.

3. DATA CLASSIFICATION

  | Level | Description | Examples | Controls |
  |-------|-------------|----------|----------|
  | Restricted | Health/special personal information | Patient records, diagnoses, test results | Encrypted, RBAC, audit logged, MFA |
  | Confidential | Personal information (non-health) | Contact details, billing, staff records | Encrypted, RBAC, audit logged |
  | Internal | Business information | Analytics, system configs, internal comms | Access controlled, standard auth |
  | Public | Published information | Marketing materials, public docs | No restrictions |

4. ACCESS CONTROL
  - Role-Based Access Control (RBAC) enforced at application and
    database level
  - Row-Level Security (RLS) in Supabase — practices can only access
    their own data
  - Principle of least privilege — minimum access required for function
  - Multi-factor authentication (MFA) available for all users
  - Session management with secure cookies (httpOnly, secure, sameSite)
  - Password requirements: minimum 8 characters, complexity enforced
  - Account lockout after 5 failed attempts

5. ENCRYPTION
  - Data at rest: AES-256 (Supabase managed encryption)
  - Data in transit: TLS 1.2+ (enforced on all connections)
  - Database connections: SSL required
  - API keys and secrets: stored in Vercel environment variables
    (encrypted, never in source code)

6. NETWORK SECURITY
  - Vercel Edge Network with automatic DDoS protection
  - Vercel Web Application Firewall (WAF)
  - Rate limiting on all API endpoints
  - API key authentication for external integrations
  - No direct database access from public internet

7. APPLICATION SECURITY
  - Input validation on all user inputs (Zod schema validation)
  - SQL injection prevention (parameterised queries via Supabase client)
  - XSS prevention (React auto-escaping, Content Security Policy)
  - CSRF protection (SameSite cookies)
  - Security headers configured (X-Frame-Options, X-Content-Type-Options)
  - Dependency scanning for known vulnerabilities

8. MONITORING AND LOGGING
  - Sentry error monitoring (real-time alerting)
  - Vercel observability (function logs, traces, analytics)
  - Audit logging for sensitive operations (data access, modifications)
  - Log retention: 12 months minimum
  - Anomaly detection for unusual access patterns

9. BACKUP AND RECOVERY
  - Automated daily backups (Supabase)
  - Point-in-time recovery capability
  - Backup encryption at rest
  - Recovery Time Objective (RTO): 4 hours
  - Recovery Point Objective (RPO): 1 hour
  - Annual disaster recovery testing

10. VULNERABILITY MANAGEMENT
  - Regular dependency updates (automated via Dependabot/Renovate)
  - Annual penetration testing (planned)
  - Responsible disclosure policy
  - Patch management: critical vulnerabilities within 24 hours,
    high within 7 days, medium within 30 days

11. PERSONNEL SECURITY
  - Background checks for employees with data access
  - Confidentiality agreements (NDA) signed on employment
  - POPIA training within 30 days of joining
  - Annual security awareness training
  - Access revocation within 24 hours of termination

12. INCIDENT RESPONSE
  See separate Incident Response Plan.

13. THIRD-PARTY MANAGEMENT
  - Sub-processors documented and authorised per Operator Agreements
  - DPAs in place with all sub-processors
  - Annual review of sub-processor security posture
  - Sub-processor list: Supabase, Vercel, Anthropic, Google, Yoco,
    Sentry, Resend

14. PHYSICAL SECURITY
  - No on-premise servers (fully cloud-based)
  - Employee devices: full-disk encryption required
  - Screen lock policy: 5 minutes inactivity
  - Clean desk policy for any physical documents

15. REVIEW
  Reviewed annually or after any security incident.
```

### 4.4 Incident Response Plan

```
VISIOCORP INCIDENT RESPONSE PLAN
Version: 1.0
Classification: Confidential

1. PURPOSE
To define procedures for detecting, responding to, and recovering from
security incidents, including security compromises as defined in POPIA
section 22.

2. INCIDENT CLASSIFICATION

  | Severity | Description | Response Time | Examples |
  |----------|-------------|---------------|---------|
  | P1 Critical | Confirmed data breach, system compromise | Immediate | Unauthorised access to patient data, ransomware |
  | P2 High | Suspected breach, major vulnerability | 1 hour | Unusual data access patterns, zero-day exploit |
  | P3 Medium | Minor security event, contained | 4 hours | Failed brute-force, phishing attempt blocked |
  | P4 Low | Security improvement needed | 24 hours | Configuration weakness, policy gap |

3. INCIDENT RESPONSE TEAM

  | Role | Primary | Backup |
  |------|---------|--------|
  | Incident Commander | [CEO/CTO Name] | [Backup Name] |
  | Technical Lead | [Lead Engineer Name] | [Backup Name] |
  | Communications Lead | [Name] | [Backup Name] |
  | Legal/Compliance | [Information Officer Name] | [Backup Name] |

4. RESPONSE PHASES

  Phase 1: DETECTION & TRIAGE (0-1 hour)
  - Confirm the incident is real (not false positive)
  - Classify severity (P1-P4)
  - Activate Incident Response Team for P1/P2
  - Document: what happened, when, what systems affected
  - Preserve evidence (logs, screenshots, system state)

  Phase 2: CONTAINMENT (1-4 hours)
  - Isolate affected systems/accounts
  - Block malicious access (IP blocks, credential rotation)
  - Prevent further data loss
  - DO NOT destroy evidence
  - DO NOT communicate externally without approval

  Phase 3: NOTIFICATION (within 24 hours)
  For P1/P2 incidents involving personal information:

  Step 1: Notify affected healthcare practices (Responsible Parties)
  - Within 24 hours of confirmation
  - Include: description, scope, impact, remediation steps
  - Channel: email to Information Officer + phone call

  Step 2: Practices notify Information Regulator
  - Within 72 hours (practice responsibility, we assist)
  - Via eServices portal: https://eservices.inforegulator.org.za
  - Using prescribed SCN1 form

  Step 3: Practices notify affected data subjects
  - In writing (email, SMS, or as directed by Regulator)
  - Include: consequences, remediation measures, recommendations

  Phase 4: ERADICATION (24-72 hours)
  - Remove root cause (malware, vulnerability, compromised credentials)
  - Patch affected systems
  - Verify fix is effective
  - Restore from clean backups if needed

  Phase 5: RECOVERY (72 hours - 1 week)
  - Restore normal operations
  - Monitor for recurrence
  - Validate data integrity
  - Confirm all systems functioning correctly

  Phase 6: POST-INCIDENT REVIEW (within 2 weeks)
  - Root cause analysis
  - Timeline reconstruction
  - Lessons learned
  - Update security measures
  - Update this plan if needed
  - Report to affected practices

5. RECORDS
  Maintain an incident register documenting:
  - Date and time of detection
  - Description of incident
  - Severity classification
  - Systems and data affected
  - Number of data subjects affected
  - Actions taken at each phase
  - Notifications issued
  - Root cause analysis
  - Remediation actions
  - Lessons learned

6. TESTING
  - Tabletop exercises: twice per year
  - Full simulation: annually
  - Results documented and shared with enterprise clients on request
```

### 4.6 Data Breach Notification Procedure

```
VISIOCORP DATA BREACH NOTIFICATION PROCEDURE
Version: 1.0
In compliance with POPIA Section 22

1. DEFINITION
A "security compromise" (data breach) occurs when there are reasonable
grounds to believe that personal information has been accessed or
acquired by any unauthorised person.

2. DETECTION CHANNELS
  - Sentry error monitoring alerts
  - Unusual access pattern detection
  - User/practice reports
  - Third-party notifications (sub-processors)
  - Penetration testing findings
  - Law enforcement notification

3. INTERNAL ESCALATION (within 1 hour)
  Step 1: Person detecting → Information Officer (immediate)
  Step 2: Information Officer assesses: Is this a confirmed breach?
  Step 3: If confirmed/suspected → activate Incident Response Plan
  Step 4: Document in incident register

4. RESPONSIBLE PARTY NOTIFICATION (within 24 hours)
  Notify each affected healthcare practice:
  - Method: Email to their Information Officer + phone call
  - Content:
    (a) Nature of the breach
    (b) Categories and approximate number of patients affected
    (c) Categories of personal information compromised
    (d) VisioCorp contact point for queries
    (e) Likely consequences
    (f) Measures taken to address the breach

5. INFORMATION REGULATOR NOTIFICATION (within 72 hours)
  The healthcare practice (Responsible Party) is responsible for
  notifying the Regulator. VisioCorp assists by:
  - Providing all information needed for the SCN1 form
  - Assisting with portal submission if requested
  - Portal: https://eservices.inforegulator.org.za

  SCN1 form requires:
  - Responsible party and information officer details
  - Date incident occurred and date reported
  - Breach classification
  - Incident description
  - Categories of compromised information
  - Number of affected data subjects
  - How data subjects were notified
  - Signed declaration of accuracy

6. DATA SUBJECT NOTIFICATION (as directed by practice)
  The practice decides on data subject notification. If requested,
  VisioCorp assists with:
  - Generating affected patient lists
  - Sending notifications via platform (email/SMS)
  - Providing recommended protective measures

7. RECORDS
  All breaches (including near-misses) are recorded in the breach
  register with:
  - Date and time
  - Description
  - Systems affected
  - Data subjects affected
  - Actions taken
  - Notifications sent
  - Root cause
  - Remediation

8. NO EXTERNAL COMMUNICATION WITHOUT APPROVAL
  VisioCorp shall not communicate with data subjects, the Information
  Regulator, or media about any breach without the Responsible Party's
  prior written consent.
```
